# 项目待补充清单：代码、硬件、模型训练与工程化

基于源项目 `project1/demo_multhread_decode_infer_mulmodel` 和现有理解笔记整理。

## 1. 代码层面目前缺少/需要补充的内容

### 1.1 工程结构与配置

- 建立干净的新工程结构，避免直接把源项目的 `build/`、第三方库、demo 文件混在一起。
- 用统一配置文件替代源码硬编码：
  - 视频源地址、数量、账号密码
  - 模型路径、类别名、模型输入尺寸、阈值
  - 输出方式：RTMP/RTSP/本地保存
  - 拼图布局、分辨率、帧率、码率
  - NPU core 绑定策略
- 增加命令行参数解析，当前 `argv[1]` 没判断，直接 `std::stoi(argv[1])` 容易崩溃。
- 增加配置合法性校验，例如路数不能超过布局容量，模型文件必须存在，推流地址不能为空。

### 1.2 主流程与线程生命周期

- 当前多个线程基本依赖 `while(true)` 或无限循环，缺少完整退出机制。
- `combineImage()` 没有退出条件，导致后面的 `stopStreaming()`、`unload_stream()`、线程 join 基本不会正常执行。
- 需要统一的全局运行状态，例如 `std::atomic<bool> running`，支持 Ctrl+C/SIGINT 优雅退出。
- 需要保证所有线程都能停止：拉流线程、解码线程、推理线程、拼图线程、推流线程。
- 需要处理模型对象释放，目前 `new rknn_lite(...)` 后没有清晰 delete/智能指针管理。

### 1.3 视频输入与解码

- RTSP 断流重连需要完善：重连间隔、最大重试、长期断流占位画面。
- 本地文件循环播放/按原始 FPS 播放需要做成配置。
- 摄像头 H.264/H.265、分辨率、帧率、编码格式兼容性需要统一处理。
- 解码输出格式、stride、对齐、颜色空间转换需要在 RK3588 上实测确认。

### 1.4 推理与模型管理

- 当前 tired 疲劳模型只预留，主流程里 `ptr3 = nullptr`，没有真正接入。
- 需要做模型统一管理：加载、释放、输入尺寸、类别数、模型 id、NPU core、阈值。
- 需要为每个模型建立清晰后处理逻辑，不能所有模型都默认套同一套 YOLO 后处理。
- 需要明确类别含义：
  - person：人员
  - helmet：安全帽/未戴安全帽？
  - callplay：打电话/玩手机？
  - tired：疲劳/闭眼/打哈欠？
- 需要支持动态启停某个模型，否则后期测试不方便。

### 1.5 结果融合与业务规则

- 现在有 DetectionFusionManager，但业务语义仍需完善：
  - 人和安全帽如何关联？
  - 没戴安全帽是否要生成告警？
  - 手机/打电话是否必须绑定到某个人？
  - 疲劳检测是基于人脸还是整个人？
- 需要补充告警规则：持续多少帧才报警、置信度阈值、同一目标去重、冷却时间。
- 需要增加事件输出结构，而不仅是画框，例如 JSON：时间、通道、类别、置信度、截图路径。

### 1.6 拼图与渲染

- 当前拼图画布是 `1280×1080`，但推流配置是 `1280×720`，尺寸不一致，会被二次 resize，影响画质和比例。
- `images(6)`、`mutexes(6)` 写死最多 6 路，需要改成按配置动态创建。
- `combineImage()` 中 `combinedImage` 没有每轮清空，如果某路断流，旧画面可能残留。
- 需要支持 1/2/4/6/9/16 路布局，按配置自动计算 tile 大小。
- 需要在画面上叠加通道名、FPS、时间戳、断流状态、告警状态。

### 1.7 编码与推流

- RTMP 目前较完整，但地址写死。
- RTSP 输出接口有预留，但需要真正实现或明确暂不支持。
- 需要处理推流服务器断开后的重连。
- 需要支持本地录像保存，便于调试和证据留存。
- 需要明确编码参数：H.264/H.265、码率、GOP、profile、CBR/VBR。
- MPP 编码输出和 FFmpeg 封装时间戳需要实测，防止播放端卡顿/倍速/音画时间基异常。

### 1.8 日志、状态与可维护性

- 增加日志系统：INFO/WARN/ERROR/DEBUG，输出到控制台和文件。
- 增加运行状态统计：每路 FPS、解码延迟、推理耗时、NPU 占用、队列长度、丢帧数。
- 增加健康检查接口或状态文件，方便现场部署排错。
- 增加异常保护，避免一个摄像头/一个模型异常导致整个程序崩溃。
- 增加 README、编译文档、运行文档、部署文档。

### 1.9 测试与验证

- 增加本地 mp4 测试模式。
- 增加模拟 RTSP 输入测试。
- 增加模型输出正确性测试。
- 增加长时间稳定性测试：至少 24h/72h。
- 增加多路压力测试：2/4/6/8 路，记录 FPS、CPU、NPU、内存、温度。

## 2. 硬件方面需要准备

### 2.1 核心计算平台

- RK3588/RK3588S 开发板或工控机，建议：
  - 内存 8GB 起步，推荐 16GB
  - 有 NPU 可用
  - Linux 系统支持 RKNN Runtime、MPP、RGA
  - 稳定电源和散热
- 存储：
  - eMMC/SSD/高速 TF 卡
  - 推荐预留 64GB 以上，如果要录像建议更大

### 2.2 摄像头与视频源

- RTSP 网络摄像头若干，建议先准备 1~2 路验证，再扩展到 4/6/8 路。
- 摄像头建议支持：
  - H.264 或 H.265
  - 1080p/720p 可调
  - 固定帧率 15/20/25fps
  - RTSP 地址可配置
- PoE 摄像头则需要 PoE 交换机。
- 非 PoE 摄像头需要单独供电。

### 2.3 网络与推流环境

- 千兆交换机/路由器，保证多路视频流稳定。
- 如果要 RTMP 推流，需要准备 RTMP 服务器，例如：
  - nginx-rtmp
  - SRS
  - EasyDarwin/ZLMediaKit 等流媒体服务
- 播放端：VLC、ffplay、网页播放器或业务平台。

### 2.4 调试设备

- HDMI 显示器或串口工具，用于板端调试。
- USB 键鼠或 SSH 网络调试环境。
- Windows 开发机/Ubuntu 开发机。
- 网线、交换机、稳定电源、散热片/风扇。

### 2.5 散热与稳定性

- RK3588 多路解码 + NPU + 编码负载下会发热，必须准备主动散热。
- 建议监控温度和降频状态。
- 现场部署建议使用稳定电源，避免低压导致异常重启。

## 3. 模型训练方面需要怎么做

### 3.1 先明确检测任务和类别

需要先确定最终业务类别，例如：

- 人员检测：person
- 安全帽检测：helmet / no_helmet
- 打电话：phone_call
- 玩手机：playing_phone
- 疲劳：tired / normal，或者 eye_closed / yawn

建议不要一开始就训练一个大而全模型。更稳的路线是：

1. 先用现有 RKNN 模型跑通全流程。
2. 再分别优化 person、helmet、callplay、tired。
3. 最后评估是否合并为一个多类别模型。

### 3.2 数据采集

- 从实际摄像头场景采集视频，抽帧成图片。
- 覆盖不同：
  - 白天/夜晚
  - 室内/室外
  - 远景/近景
  - 遮挡
  - 低清晰度
  - 不同角度
  - 不同安全帽颜色
  - 不同手机姿态
- 每个类别建议至少几千张有效标注图；复杂场景越多越好。

### 3.3 数据标注

- 目标检测建议用 YOLO 格式标注。
- 工具可用：LabelImg、CVAT、Roboflow、Label Studio。
- 标注重点：
  - 框要贴合目标
  - 类别定义不能混乱
  - helmet/no_helmet、call/play_phone 等类别必须标准统一
- 数据划分：train/val/test，常见比例 7:2:1 或 8:1:1。

### 3.4 模型训练

推荐路线：

- PC/GPU 上训练 YOLO 系列模型，例如 YOLOv5/YOLOv8/YOLO11 的 n/s 小模型。
- 输入尺寸建议先用 640×640。
- 训练后查看 mAP、Precision、Recall、混淆矩阵。
- 根据误检漏检补数据，而不是只调参数。

### 3.5 导出与 RKNN 转换

常见流程：

```text
PyTorch .pt
  ↓ export
ONNX
  ↓ rknn-toolkit2 转换/量化
RKNN
  ↓ RK3588 板端 RKNN Runtime 推理
```

需要准备：

- rknn-toolkit2，通常在 x86 Ubuntu/WSL/容器中转换。
- 量化校准数据集，建议使用真实场景图片 200~1000 张。
- 确认 target platform：RK3588。
- 转换后在板端验证输出是否和 ONNX/PyTorch 对齐。

### 3.6 后处理适配

这是很容易被忽略的一点：

- 不同 YOLO 版本输出格式不一样。
- RKNN 输出张量顺序、量化类型、坐标解码方式可能不同。
- 当前 `postprocess.cpp` 不一定适配你重新训练的模型。
- 每换模型，都必须确认：输入尺寸、归一化、通道顺序、输出层、anchor/stride、NMS 逻辑。

### 3.7 模型评估指标

不仅看 mAP，还要看：

- RK3588 实际推理耗时
- 多路并发时 FPS
- 漏检率/误检率
- 小目标表现
- 夜间/模糊/遮挡表现
- NPU core 利用率
- 量化后精度下降情况
